End-to-end data flow

Top row = setup. Middle row = field ops. Bottom row = post-field exports + live Hub view.
Phase 1
Pre-field prep — done at the desk by a breeder or admin once per nursery.
1 Initialise from PRISM export Admin panel · Step 1

Upload the PRISM "Nursery site" Excel export. The app reads every row, peeks at any embedded Map sheet to figure out spike groupings, and stamps every packet with a unique 8-char UUID + QR payload of the form SNUR:<nursery>:<uuid>.

Inputs: Nursery code, PRISM export .xlsx, sheet name (auto-detected). Outputs: 1,300+ packets seeded in SQLite, a 1.8 MB QR-coded PDF of packet labels, and a 14-tab workbook (<nursery>_workbook.xlsx) for reference.

2 Build Nursery list Admin panel · Step 2

Groups every packet by Source ID, counts repeats, and multiplies by 1.4 to get the required seed quantity per packet. Source IDs with more than 10 reps are flagged BULK for the bulk-treatment line.

3 Design Field Map Admin panel · Field Map editor

The Field Map editor lets you redefine spike-to-row mappings (e.g. Spike 1 = rows 1–10, Spike 2 = rows 11–26) and saves them to the field_maps table. Saving triggers a recompute that updates spike + rack_order for every packet in this nursery.

4 Generate Packet PDF Admin panel · Download

A branded PDF (4 columns × 8 rows per A4 page) with one QR-coded label per packet. Header band shows nursery code + page number. Each label has plot, spike, rack-order, material ID, source ID, generation, CMS, and the QR payload printed as text underneath (fallback if scanner misreads).

5 Sort packets for racking (LSD Radix) Auto · in workbook + manifest

Packets are physically picked in Rack Order (ascending) within each Spike (ascending) — the standard Least-Significant-Digit radix sort used in the original VBA. This is the order they should be stacked in racks for planting.

Phase 2
Field operations — done on a phone in the paddock by techs.
6 Record replacement PWA · 🔄 Replacement tile

Tech scans the packet QR (or finds the packet by plot if the QR is damaged), taps the Replacement tile, enters the stage (Packeting/Planting), the original Source ID, and the replaced-with Source ID. Saved to IndexedDB outbox immediately; syncs to the backend as soon as the device has connectivity.

Offline-first. All event tiles save instantly to IndexedDB on the phone. The sync to the backend is a background concern — if the tech steps out of Wi-Fi range, events queue up and flush when they walk back.
Idempotent. Each event has a client-generated UUID. The /sync endpoint refuses duplicates, so retries are safe.
7 Record planting error PWA · ⚠️ Planting error tile

Pick from a preset list of error kinds (Missed plot, Wrong packet planted, Damaged packet, Mixed packets, Other), set severity, and add a free-text note. Same event-capture pipeline as Replacement — single source of truth.

8 Record spray applied PWA · 💧 Spray applied tile

Pick product (TFMSA / IMI / HPPD / Other), date (defaults to today), and optional rate. One event per packet per spray pass.

9 Record AB bag pulling PWA · 📦 AB bag pulled tile

Enter bag count + date pulled. Used during weekly AB-line management.

Phase 3
Post-field & sync — produces the Fieldbook + live Hub view.
10 Pull updated PRISM Admin panel · Step 1 (re-run)

Once the breeder updates PRISM with replacements and errors, download the refreshed Nursery site .xlsx and re-run Step 1 with it. Existing packets get refreshed; field events (already in SQLite) are preserved.

11 Generate Fieldbook Admin panel · Download Fieldbook .xlsx

Pulls every event for the nursery from SQLite, groups by event type, and writes a multi-tab Excel workbook (Replacements, Planting errors, Spray log, AB bag pulling, Date recording, Notes). Each sheet has Plot, Material, Source, Tech, Captured-at, plus type-specific columns. Hand to the breeder for PRISM upload.

12 Refresh dashboard /dashboard route · live

The /dashboard page is server-rendered on every page load — shows every registered nursery, packet counts, event counts by type and tech, and a hyperlinked file path to open each underlying workbook. No manual refresh step; just navigate there.

13 Push to Hub Continuous · via /sync

Unlike the Excel workflow, there's no separate batch push. Every /sync call from the PWA IS the push — the Hub view always reflects the live state. The cycle ends here; you're ready to start the next season or onboard another nursery.
